Qu'est-ce que PlayPad ?

PlayPad est une application web qui permet de gérer sa bibliothèque de jeux vidéo toutes plateformes confondues (Steam, PlayStation, Xbox, Nintendo, Epic Games).

Elle offre des fonctionnalités sociales : ajouter des amis, voir leur bibliothèque, leur proposer des jeux, poster des critiques publiques, et constituer un Top 3.

Les jeux Steam sont importés automatiquement via l'API Steam (avec les succès). Un catalogue partagé permet de découvrir des jeux et de les ajouter en un clic.

Sécurité

Protections mises en place pour sécuriser l'application :

Helmet

Headers HTTP sécurisés : HSTS, X-Content-Type-Options, X-Frame-Options, etc. Protège contre les attaques XSS, clickjacking, MIME sniffing.

Rate Limiting

200 requêtes / 15 min (global). 10 tentatives / 15 min sur les routes d'auth (login, register, reset password). Protège contre le brute-force et le spam.

Admin Reset sécurisé

La route /api/admin/reset est maintenant protégée par une liste d'IDs admin (variable ADMIN_IDS). Sans cette variable, tous les users authentifiés ont accès (mode dév).

Cookie Secure corrigé

Le flag secure du cookie de session désormais dépend uniquement de NODE_ENV=production. Auparavant, PUBLIC_URL déclenchait le mode secure, cassant la connexion en HTTP local.

Endpoint Debug protégé

/api/debug/env nécessite maintenant l'authentification + droits admin. Exposait précédemment les variables d'env à tous les visiteurs.

Sync limité

L'import de jeux est limité à 5000 jeux par requête pour éviter les abus mémoire et les attaques par dénion de service.

Architecture

L'application suit une architecture classique client-serveur :

Navigateur
  ↓ HTTPS
Express (Node.js)  ← Render.com
  ↓ Service Role Key
Supabase (PostgreSQL)  ← Base de données

Le frontend est une Single Page Application React (UMD, sans build). Le backend Express expose une API REST. Les sessions sont stockées dans Supabase pour persister les connexions.

Stack technique

Frontend

  • React 18 (UMD, sans build)
  • Tailwind CSS (CDN)
  • Icônes SVG inline
  • Pas de dépendances npm frontend

Backend

  • Express.js
  • express-session + SupabaseStore
  • bcryptjs (hash mots de passe)
  • @supabase/supabase-js
  • helmet (sécurité headers HTTP)
  • express-rate-limit (protection brute-force)

Base de données

  • Supabase (PostgreSQL)
  • 9 tables + 2 index
  • Session store personnalisé

Hébergement

  • Render.com (Web Service)
  • Node.js 18+
  • Déploiement Git automatisé

Flux de données

1. Utilisateur ouvre l'application → index.html chargé (SSR React)
2. React monte → loadData() appelé
3. GET /api/me → vérification session
4. GET /api/games, /api/wishlist, /api/topthree
5. Toutes les 120s : POST /api/heartbeat
6. Actions utilisateur → POST/PUT/DELETE mutent la DB via Supabase

Fonctionnalité clé : Connexion Steam

  1. L'utilisateur clique "Connecter Steam" (lien OpenID dans le panneau sync)
  2. Le serveur redirige vers steamcommunity.com/openid/login
  3. L'utilisateur s'authentifie sur Steam (connexion officielle, pas de mot de passe partagé)
  4. Steam redirige vers le callback /api/auth/steam/callback
  5. Le serveur valide la réponse OpenID via check_authentication
  6. Extraction du Steam ID depuis openid.claimed_id
  7. Appel à l'API Steam (GetOwnedGames) pour récupérer les jeux
  8. Pour chaque jeu : appel à GetPlayerAchievements pour les succès
  9. Sauvegarde des jeux + ajout au catalogue partagé

Fonctionnalité clé : Sessions persistantes

Les sessions sont stockées dans la table sessions de Supabase via un store personnalisé (session-store.js).

// session-store.js — stocke les sessions dans Supabase
get(sid)   → SELECT session_data FROM sessions WHERE sid = ?
set(sid, session) → UPSERT INTO sessions (sid, session_data, expires)
destroy(sid) → DELETE FROM sessions WHERE sid = ?

Avantage : les sessions survivent aux redémarrages de Render. Fallback mémoire si la table n'existe pas encore.

Installation et configuration

Variables d'environnement

SUPABASE_URL            = https://xxx.supabase.co
SUPABASE_ANON_KEY       = (clé anon du projet Supabase)
SUPABASE_SERVICE_ROLE_KEY = (clé service_role, pour les opérations admin)
STEAM_API_KEY           = (clé API Steam, optionnelle pour l'import)
XBL_API_KEY             = (clé API Xbox via xbl.io, optionnelle)
PUBLIC_URL              = https://ton-app.render.com (pour le callback Steam)
SESSION_SECRET          = (optionnel, pour les cookies de session)
ADMIN_IDS               = (optionnel, IDs séparés par des virgules, ex: "1,3")

ADMIN_IDS : si défini, seuls ces utilisateurs peuvent appeler /api/admin/reset et /api/debug/env. Si vide, tous les users authentifiés y ont accès (mode développement).

Base de données Supabase

Exécuter le fichier supabase-schema.sql dans l'éditeur SQL de Supabase :

Dashboard → SQL Editor → Coller le contenu du fichier → Run

Ce script crée toutes les tables nécessaires (CREATE TABLE IF NOT EXISTS, exécutable plusieurs fois sans risque).

Déploiement sur Render

  1. Créer un Web Service sur Render.com
  2. Brancher le repository GitHub
  3. Build Command : yarn
  4. Start Command : node Site/server/server.js
  5. Ajouter les variables d'environnement

Playnite Bridge (import local)

Le dossier playnite-bridge/ contient un script Node.js qui lit la base SQLite de Playnite et pousse les jeux vers le serveur PlayPad.

cd playnite-bridge
yarn install
# Pousser les jeux directement vers Render :
PLAYPAD_URL=https://ton-app.render.com \
PLAYPAD_USER=mon_login PLAYPAD_PASS=mon_mdp \
node playnite-reader.js --push
